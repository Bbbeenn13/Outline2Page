# Outline2Page 项目级指令

本文件优先级高于上层 monorepo 指令；若与上层规则冲突，以本文件为准。

## 语言与风格

- 所有回复使用中文。
- 代码注释使用中文。
- Commit message 使用英文 conventional commits。
- 不引入不必要的依赖。

## 需求澄清规则

整个开发过程中，如果用户意图、业务规则、验收标准、交互细节、数据含义或实现边界不明确，必须主动向用户提问确认。

禁止在关键信息不清楚时随意判断用户意图，尤其不要擅自决定：

- 用户真正想修的是哪个问题。
- 某个字段、组件、按钮、页面或流程的业务含义。
- 多种可行方案中哪一种才符合用户预期。
- 是否可以改变已有交互、视觉、数据结构或输出格式。
- 是否可以把某些问题暂时忽略、绕过或降级处理。

提问要求：

- 问题要具体，能直接影响实现决策。
- 能用本地代码、文档或现有测试确认的，不要反复问用户，先自己查清楚。
- 如果存在多个解释，先列出差异和影响，再请用户确认。
- 在用户确认前，不要进入会让方案定型的代码实现。

## Fix 工作流强制规则

先按问题性质判断阶段，不要只根据用户是否说了 `fix`、`修复`、`bugfix` 机械进入 Phase 2。

- 影响核心流程、生成结果、数据结构、模板写入，或需要排查根因的问题：进入 `docs/phase2.Fix Problem/`，必须使用 `structured-fix-workflow` skill。
- UI polish、小体验调整、文案微调、非阻断性局部改动：进入 `docs/phase3.Patch/`，按该目录 README 的轻量规则记录。
- 边界不清楚时，先说明 Fix/Patch 差异并向用户确认。

执行 fix 时必须遵守：

1. 先读取项目指令、现有 fix 文档、Git 状态和相关代码，再动手。
2. fix 文档固定写入 `docs/phase2.Fix Problem/<version>/`，禁止使用 `doc/fix/<version>/` 或 `docs/2.Fix Problem/<version>/`。
3. `<version>` 使用最新版本规则：
   - 如果是在继续当前未完成 fix，沿用当前最高版本目录。
   - 如果上一轮 fix 已交付或已验证，新建下一个版本目录，例如 `v3.0` 之后使用 `v4.0`。
   - 不得把新需求、新决策、新进度混入旧版本历史记录。
4. 每个 fix 版本目录必须包含：
   - `proposal.md`
   - `task.md`
   - `progress.md`
   - `debrief.md`
   同时必须维护总复盘文件 `docs/phase2.Fix Problem/RETRO.md`。
5. `proposal.md` 必须记录背景、当前问题、目标方案、不做范围、待确认问题、验收标准、风险。
6. `task.md` 必须把修复拆成可执行任务，每个任务包含目标、涉及文件、完成标准、测试要求。
7. `progress.md` 必须持续记录当前阶段、基线状态、检查清单、问题决策、风险、下一步、验证结果。
8. `debrief.md` 必须在本轮修复收尾时输出，内容面向下一轮修复复盘，至少包含：
   - 本轮问题根因
   - 哪些判断是正确的
   - 哪些判断走偏了
   - 哪些测试或检查最有效
   - 下次遇到同类问题的操作准则
   - 仍需注意的风险
9. 每轮 fix 完成 `debrief.md` 后，必须把跨版本可复用的教训、避坑规则、必测项或长期风险同步合并进 `docs/phase2.Fix Problem/RETRO.md`。`RETRO.md` 只记录沉淀后的高价值结论，不复制完整修复过程。
10. 进入代码实现前，如果需求会影响实现形状且不明确，必须先把问题写入 `proposal.md` 和 `progress.md`，并向用户确认。
11. 实现完成后必须运行项目可用的完整验证命令；缺失的命令要明确记录，不得假装已验证。

## 删除限制

禁止批量删除文件或目录。不要使用：

- `del /s`
- `rd /s`
- `rmdir /s`
- `Remove-Item -Recurse`
- `rm -rf`

需要删除文件时，只能一次删除一个明确路径的文件。如果需要批量删除文件，必须停止操作并请用户手动删除。
