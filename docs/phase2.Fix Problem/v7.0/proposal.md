# v7.0 TOC title 注入失效重新审视方案

## 背景

用户反馈 v5.0 与 v6.0 均未解决 TOC 页 title 注入问题，并指出当前结果像固定读了旧槽位内容或缓存，不会随大纲变化而调整。

## 当前问题

- v5.0 与 v6.0 的修复有效性需要重新审视，不能继续把它们的结论当作可信前提。
- 真实表现不只是 clone 数量问题，还包括 title 文本不随输入大纲变化。
- 截图显示真实 `TOC_NAV_group` 内部通过多个 `NAV_item` 变体暴露 `HIGHLIGHT` 与 `TOC_*_TEXT`，这可能要求写入文本时同步切换可见状态/变体，而不是只写文本属性。
- 也需要确认生成链路是否每次从模板重新 clone，而不是复用旧 generated 节点或旧 property override。

## 目标方案

- 先用代码与测试重新验证 v5.0/v6.0 的判断边界，标记不可靠沉淀。
- 复核生成链路：模板 clone、旧生成 Section 清理、属性写入顺序、TOC 专属注入是否覆盖普通注入。
- 复核 TOC 写入链路：`TYPE`、`HIGHLIGHT`、`SHOW`、`TOC_TITLE_TEXT` 在真实模板中的配合关系。
- 建立能复现“文本写入固定旧值、不随大纲变化”的回归测试，再做修复。

## 不做范围

- 不根据 v5.0/v6.0 结论继续假设真实模板结构。
- 不改变大纲语法。
- 不引入不必要依赖。

## 待确认问题

暂不向用户提问。当前先从本地代码、已有截图和可构造的 Figma 属性模型里排查；若最终必须知道真实 raw property key，再请求用户扫描结果或调试输出。

## 验收标准

- 能解释为什么旧 title 内容像缓存一样保留。
- 生成同一个 TOC 模板时，换不同大纲能写入不同 `TOC_TITLE_TEXT`。
- 目录项数量和显示文本随章节/标题数据变化。
- v5.0/v6.0 的错误或不完整结论被写入 debrief，并同步修正 RETRO。
- 完整验证命令通过。

## 已知风险

- 没有直接访问 Figma runtime 的真实 raw property key，只能用测试 fixture 模拟；如 fixture 与真实 key 仍不一致，需要用户提供扫描输出。
