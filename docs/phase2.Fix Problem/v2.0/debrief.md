# Outline2Page Fix 2.0 Debrief

## 本轮问题根因

Fix 2.0 的核心问题是 Fix 1.0 采用的短 frame 命名过于贫瘠。`02.chapter`、`03.title`、`04.step` 可以表达页面类型，却无法直接表达页面所在的完整大纲层级，因此依赖 frame 名称的导航解析只能知道“这是章节页、标题页或小节页”，不能从名称本身恢复章节、标题、小节路径。

用户进一步明确需要 `<page>.<chapter>/<title>/<step>` 这种完整层级路径，frame 名称既要可读，也要能为导航高亮提供页面层级判断。

## 有效判断

- 把 Fix 2.0 范围严格限制在 frame 命名和导航解析，没有顺手改 UI、属性注入或模板扫描，控制了变更面。
- 保留 `00.cover` 和 `01.toc` 的稳定短名，避免封面和目录引入无意义路径。
- 对章节、标题、小节页分别输出 `02.<chapter>`、`03.<chapter>/<title>`、`04.<chapter>/<title>/<step>`，满足用户对完整层级路径的要求。
- 导航解析同时兼容新完整路径和旧 `page.kind` 短格式，降低旧文件或旧测试被破坏的风险。
- frame 名无法解析时继续回退到 `PagePlanItem`，让导航高亮不会因为命名异常完全失效。

## 走偏判断

- Fix 1.0 中把 `page.kind` 短格式视为足够稳定，低估了用户对完整层级可读性的需求。
- 早期曾希望 frame 名承担更强的索引职责，但本轮最终更适合让 frame 名负责层级可读和页面类型判断，具体索引仍由 page plan 提供。
- 文档记录里仍出现过 `doc/fix/2.0` 旧路径，后续已经统一为 `docs/phase2.Fix Problem/v2.0`。

## 有效测试或检查

- `naming-service` 定向测试验证章节、标题、小节的完整路径输出，是本轮最关键的回归保护。
- `navigation-injector` 定向测试验证新 frame 名可被识别，同时兼容旧短格式和解析失败回退。
- 完整运行 `npm test`、`npm run typecheck`、`npm run lint`、`npm run build`，确认命名服务、导航解析和构建产物一致。
- 更新 `code.js` 后再验证，避免源码已修但插件入口仍是旧逻辑。

## 下次同类问题操作准则

1. 任何命名规则变更都要先确认 frame 名承担什么职责：展示给人读、机器解析页面类型、机器解析索引，还是三者都有。
2. 如果用户要求“完整层级”，不要用短类型名替代；必须保留章节、标题、小节的真实文本路径。
3. 修改 frame 命名时，同步检查所有依赖 frame 名的解析逻辑，尤其是导航高亮。
4. 新命名规则上线时，至少保留一轮旧格式兼容，避免旧生成结果和测试夹具断裂。
5. `/` 在 Figma 图层名里是普通字符，但外部导出工具可能把它当路径分隔符；如果未来接入导出链路，需要重新评估转义策略。
6. frame 名可以判断页面层级，但具体 chapter/title/step 索引仍应优先使用结构化 page plan，避免同名标题导致歧义。

## 剩余风险

- 大纲真实文本中如果本身包含 `/`，可能让 frame 名的视觉路径产生歧义；后续可考虑只在解析层做更稳健的转义或分段记录。
- 同名章节、同名标题或同名小节会降低仅凭 frame 名定位索引的可靠性，因此索引仍不能完全依赖名称文本。
- 如果未来用户需要通过 frame 名直接反查精确索引，需要新增去重策略或隐藏机器标识，而不是继续扩展当前可读名称。
- v2 文档历史中残留旧路径 `doc/fix/2.0`，以后只应使用 `docs/phase2.Fix Problem/v2.0`。
