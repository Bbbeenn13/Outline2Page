# v5.0 修复复盘

## 本轮问题根因

- TOC 标题扩展只尝试 clone 内部 `TOC_TITLE_TEXT` 实例；真实模板里该实例可能位于嵌套组件内部，clone 调用会失败。
- 普通属性注入器跳过导航容器后，仍按全量默认目标计算缺失属性，导致导航专属属性被误报。

## 有效判断

- 先把 `TOC_NAV_group` 继续视为容器，标题写入目标从内部 `NAV_item/TOC_TITLE_TEXT#...` 收集。
- 对嵌套标题槽位增加失败复现测试，再实现逐级 clone 回退。
- 缺失属性检查只统计普通注入器实际负责过的目标，避免吞掉真实内容缺失。

## 走偏判断

- 最初实现只找第一个有 `clone` 的节点，忽略了真实 Figma 中“有 clone 但调用失败”的情况；后续改为逐级尝试可复制候选。

## 有效测试或检查

- `toc-expander` 新增内部标题实例不可 clone、父级标题行可 clone 的回归测试。
- `property-injector` 新增跳过导航组时不产生 missingProperties 的断言。
- 完整验证覆盖 `npm test`、`npm run typecheck`、`npm run lint`、`npm run build`。

## 下次同类问题操作准则

- 嵌套 Figma 组件槽位扩展不能只 clone 目标实例；应沿父级链寻找可复制且可追加的最小容器。
- 专属注入器负责的导航/目录属性，不应由普通属性注入器报告缺失。
- 新增 Figma clone 策略时必须测试 clone 抛错后的回退路径。

## 剩余风险

- 真实模板如果标题行父级也不可 clone，仍会 warning，需要模板侧提供可复制容器。
- 当前修复不改变目录布局尺寸规则；极端标题数量仍可能需要模板布局配合。
