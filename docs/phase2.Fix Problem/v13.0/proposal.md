# v13.0 字段映射生成时回退修复方案

## 背景

用户在“字段映射”输入框中手动追加属性后点击“生成”，界面会立刻回到扫描模板得到的默认配置。该问题影响用户自定义字段映射能否稳定进入生成链路，属于核心生成前配置流程问题。

## 当前问题

- `src/ui/ui-controller.ts` 中点击“生成”会先收集当前字段映射，再设置 `state.generating = true` 并调用 `render()`。
- `renderProperties()` 每次渲染都会根据扫描到的模板属性重新计算默认值，并重新写入输入框 `value`。
- 因此用户刚手动输入的值虽然已经被本次消息收集，但 UI 会显示成默认配置；如果生成失败、完成后再次生成，用户也容易丢失自定义输入。
- `src/ui.html` 内嵌脚本保留同一套逻辑，构建产物 `code.js` 也需要同步。

## 目标方案

- 在 UI 状态中保存当前字段映射文本值。
- 字段映射输入框发生 `input` 时更新状态。
- 触发生成前先同步一次当前 DOM 输入到状态，再发送 `propertyMapping`。
- `renderProperties()` 优先使用用户状态值；只有该字段尚未被用户编辑时才使用模板扫描默认值。
- 保持 `collectPropertyMapping()` 输出结构不变，避免影响主线程和属性注入器。

## 不做范围

- 不引入字段映射持久化，例如 `figma.clientStorage`。
- 不改变字段映射的分隔符规则。
- 不改变模板扫描、属性注入、TOC/PAGE_NAV 协议。
- 不改变字段映射的业务含义或默认映射列表。

## 待确认问题

- 暂无。当前问题可由现有代码和截图直接确认，修复方向不改变业务规则。

## 验收标准

- 用户在字段映射输入框中追加或修改属性后，点击“生成”不会把输入框显示回默认配置。
- 生成消息中的 `propertyMapping` 包含用户修改后的值。
- 生成失败或生成完成触发 UI 重绘后，用户输入仍保留。
- `src/ui.html`、`src/ui/ui-controller.ts`、`code.js` 行为一致。
- 项目完整验证通过：`npm test`、`npm run typecheck`、`npm run lint`、`npm run build`。

## 风险

- 测试 Fake DOM 当前通过解析 `innerHTML` 模拟输入，需要扩展以能模拟用户真实编辑。
- 如果未来需要跨插件会话保留字段映射，应另开版本设计持久化策略，不能把本轮临时 UI 状态误当成长期存储。
