# v6.0 修复复盘

## 本轮问题根因

- `TOC_TITLE_TEXT` 槽位不足时，标题扩展逻辑会从标题实例一路向父级寻找可 clone 节点。
- 当 `TOC_TITLE_TEXT` 暴露在外层 `TOC_NAV_group` 自身时，`TOC_NAV_group` 被误判为标题槽位 clone 候选。
- 结果是为了补一个标题槽位复制整张目录章节卡片，造成重复目录行、数量对不上；该复制发生在标题槽扩展阶段，所以面板的章节行 `TOC 扩展` 仍显示 0。

## 有效判断

- 从截图里的额外重复 04 卡片和 `TOC 扩展 0` 组合反推，问题不是章节行扩展，而是标题槽补位误复制。
- 保留 v5.0 的内部标题行 clone 策略，但把 clone 边界限制在当前 `TOC_NAV_group` 内部。

## 走偏判断

- 初看容易把“前三个章节标题空白”归因到纯文本注入失败；真正更危险的是标题槽扩展把外层目录行当成了标题槽位，导致结构被污染。

## 有效测试或检查

- 新增 `does not clone the whole TOC_NAV_group when row-level title properties are insufficient`，覆盖外层暴露标题属性且标题数量不足的场景。
- `npm test`、`npm run typecheck`、`npm run lint`、`npm run build` 全部通过。

## 下次同类问题操作准则

- 子槽位扩展必须明确 clone 边界；补内部标题、导航、面包屑槽位时，不能越界复制整页或整张卡片。
- 报告计数为 0 但画布多出节点时，要检查是否存在未计入报告的二级扩展逻辑。
- 外层实例暴露嵌套属性不等于外层实例就是可复制的业务槽位。

## 剩余风险

- 如果真实模板只把多个标题文本暴露在外层 `TOC_NAV_group`，且没有可复制的内部标题 `NAV_item` 或标题行，代码会给 warning，但不能自动补足不可见结构。
- 模板侧仍应保证每个需要动态扩展的标题槽位有可 clone 的内部容器。
