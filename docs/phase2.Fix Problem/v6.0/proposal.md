# v6.0 TOC 标题注入与重复目录行修复方案

## 背景

用户在真实 Figma 模板中生成目录页后，插件无报错，但目录页出现 `TOC_TITLE_TEXT` 注入失效、目录项数量与大纲不一致、前面三个章节标题区域为空，以及额外重复的章节卡片。

## 当前问题

- `TOC_TITLE_TEXT` 槽位不足时，现有逻辑会沿父级寻找可 clone 节点；如果标题属性暴露在 `TOC_NAV_group` 实例本身上，候选节点会变成整张目录章节卡片。
- clone 整个 `TOC_NAV_group` 会新增一张章节卡片，而不是新增标题槽位，导致目录数量对不上，并可能出现重复章节。
- 标题槽位扩展数量没有进入面板报告，用户看到 `TOC 扩展 0`，但画布里实际可能已经被补出了额外目录节点。
- 某些模板的标题显示依赖内部 `NAV_item` 槽位；当代码误把外层 `TOC_NAV_group` 当成标题槽位时，标题文本可能写入了不可见或错误层级。

## 目标方案

- 标题槽位扩展时禁止把 `TOC_NAV_group` 本身作为标题槽位 clone 候选，避免为了补标题而复制整张章节卡片。
- 优先只扩展 `TOC_NAV_group` 内部的标题 `NAV_item` 或其最小可复制父级。
- 当模板只把标题文本暴露在外层 `TOC_NAV_group` 上、且槽位不足时，给出明确 warning，而不是静默复制整张目录行。
- 补充回归测试覆盖“外层 `TOC_NAV_group` 暴露标题文本且标题数量不足”的真实风险场景。

## 不做范围

- 不重新设计 TOC 模板布局。
- 不修改大纲解析规则。
- 不改变页面数量、分页和章节范围算法。
- 不新增依赖。

## 待确认问题

无。截图和代码路径已能确定问题属于 TOC 模板写入/扩展链路错误，修复目标明确。

## 验收标准

- 为标题槽位扩展时不会复制整张 `TOC_NAV_group`。
- 槽位不足且无法安全扩展时产生明确 warning。
- 已有嵌套标题行可 clone 的模板仍能正常扩展标题槽位。
- TOC 回归测试、完整测试、类型检查、lint、build 全部通过。

## 已知风险

- Figma 真实模板可能同时存在外层暴露属性和内部实例属性，收集逻辑需要保持兼容。
- 如果模板完全没有可复制的内部标题槽位，代码只能 warning，模板侧仍需提供足够槽位或可复制标题行。
