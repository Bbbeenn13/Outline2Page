# Outline2Page 模块设计文档

## 1. 模块划分原则

模块之间尽量保持独立。

设计原则：

1. 纯逻辑模块不依赖 Figma API。
2. Figma 节点操作集中在适配层。
3. UI 只负责展示和收集输入。
4. 每个模块都有明确输入、输出、错误和测试点。
5. 缺失模板、缺失属性、解析异常均通过警告对象向上返回，不在底层直接中断全流程。

## 2. 模块总览

| 模块 | 职责 | 是否依赖 Figma API | 可独立测试 |
| --- | --- | --- | --- |
| OutlineParser | 解析 Markdown 大纲 | 否 | 是 |
| OutlineAnalyzer | 统计层级与所需模板 | 否 | 是 |
| TemplateScanner | 扫描当前 Page 模板 | 是 | 需要 Mock |
| TemplateMapper | 管理页面类型与模板映射 | 否 | 是 |
| PagePlanner | 生成页面计划 | 否 | 是 |
| NamingService | 生成 Frame 名称 | 否 | 是 |
| PaginationService | 计算实际页码与章节跨度 | 否 | 是 |
| LayoutEngine | 计算节点坐标 | 否 | 是 |
| GeneratedSectionManager | 覆盖旧 Section 并创建新 Section | 是 | 需要 Mock |
| NodeFactory | 根据模板生成节点 | 是 | 需要 Mock |
| PropertyInjector | 注入组件属性与 Text Layer | 是 | 需要 Mock |
| NavigationInjector | 注入导航 SHOW/HIGHLIGHT | 部分依赖 | 可拆分测试 |
| TocExpander | 扩展 NAV_group 并写目录 | 是 | 需要 Mock |
| ReportBuilder | 汇总报告 | 否 | 是 |
| PluginUiBridge | UI 与主线程通信 | 是 | 需要集成测试 |

## 3. OutlineParser

### 3.1 职责

解析用户粘贴的 Markdown，生成结构化大纲树。

### 3.2 输入

```ts
type ParseInput = {
  markdown: string;
};
```

### 3.3 输出

```ts
type ParseOutput = {
  document: OutlineDocument;
  warnings: ParseWarning[];
};
```

### 3.4 识别规则

| 输入格式 | 节点类型 |
| --- | --- |
| `《...》` | Vision |
| `# ...` | TOC |
| `## ...` | CHAPTER |
| `### ...` | TITLE |
| `##### ...` | STEP |

`####` 属于未定义层级，记录警告但不中断。

### 3.5 警告

1. 缺少 Vision。
2. 缺少 TOC。
3. TITLE 没有上级 CHAPTER。
4. STEP 没有上级 TITLE。
5. 未定义层级。
6. 空标题。

### 3.6 测试点

1. 正常解析完整层级。
2. 只有 CHAPTER。
3. CHAPTER + TITLE。
4. CHAPTER + TITLE + STEP。
5. STEP 缺少 TITLE。
6. TITLE 缺少 CHAPTER。
7. 出现 `####`。
8. Vision 含中文书名号。

## 4. OutlineAnalyzer

### 4.1 职责

从大纲树中统计生成需要的信息。

### 4.2 输入

```ts
type AnalyzeInput = {
  document: OutlineDocument;
};
```

### 4.3 输出

```ts
type OutlineSummary = {
  vision: string | null;
  chapterCount: number;
  titleCount: number;
  stepCount: number;
  estimatedPageCount: number;
  requiredPageKinds: PageKind[];
};
```

### 4.4 规则

1. 有 Vision 时需要 `COVER`。
2. 有 TOC 时需要 `TOC`。
3. 有 CHAPTER 时需要 `CHAPTER`。
4. 有 TITLE 时需要 `TITLE`。
5. 有 STEP 时需要 `STEP`。
6. 预计页数 = COVER + TOC + CHAPTER 数 + TITLE 数 + STEP 数。

### 4.5 测试点

1. 不同层级组合下 requiredPageKinds 正确。
2. 页数统计正确。
3. 缺 Vision 时不要求 COVER，还是记录警告由 Parser 负责。

## 5. TemplateScanner

### 5.1 职责

扫描当前 Figma Page 中所有以 `PAGE_TEMP：` 开头的模板节点。

### 5.2 输入

```ts
type TemplateScanInput = {
  currentPage: PageNode;
};
```

### 5.3 输出

```ts
type TemplateScanOutput = {
  templates: TemplateInfo[];
  warnings: TemplateWarning[];
};
```

### 5.4 规则

1. 只扫描当前 Page。
2. 模板必须以 `PAGE_TEMP：` 开头。
3. 模板节点保留不动。
4. 支持 Frame、Component、Instance。
5. 不可复制节点进入警告。
6. 扫描模板内部暴露的 componentProperties。
7. 扫描模板内部同名 Text Layer。

### 5.5 测试点

1. 当前 Page 中能找到模板。
2. 非当前 Page 模板不会被扫描。
3. 能提取 `PAGE_TEMP：TOC` 的 kindGuess 为 `TOC`。
4. 能扫描组件属性名。
5. 能扫描 Text Layer 名称。

## 6. TemplateMapper

### 6.1 职责

管理页面类型与用户选择模板之间的关系。

### 6.2 输入

```ts
type TemplateMappingInput = {
  requiredPageKinds: PageKind[];
  templates: TemplateInfo[];
  selectedTemplateIds: Record<PageKind, string>;
};
```

### 6.3 输出

```ts
type TemplateMappingResult = {
  mapping: Record<PageKind, TemplateInfo | null>;
  missingKinds: PageKind[];
  warnings: TemplateWarning[];
};
```

### 6.4 规则

1. 每个 requiredPageKind 都应有选择模板。
2. 缺失模板不阻止整体生成。
3. 缺失模板对应页面在 PagePlanner 或 NodeFactory 阶段跳过。

### 6.5 测试点

1. 完整映射。
2. 缺一个模板。
3. 用户选择了不存在的模板 id。

## 7. PagePlanner

### 7.1 职责

把大纲树转换为扁平页面计划。

### 7.2 输入

```ts
type PagePlannerInput = {
  document: OutlineDocument;
  templateMapping: Record<PageKind, TemplateInfo | null>;
};
```

### 7.3 输出

```ts
type PagePlannerOutput = {
  pages: PagePlanItem[];
  skippedPages: SkippedPage[];
  warnings: PlanningWarning[];
};
```

### 7.4 规则

1. Vision 生成 COVER。
2. TOC 生成 TOC。
3. 每个 CHAPTER 生成 CHAPTER。
4. 每个 TITLE 生成 TITLE。
5. 每个 STEP 生成 STEP。
6. 缺少对应模板时跳过该类型页面并记录警告。
7. 页面顺序按大纲顺序。

### 7.5 测试点

1. 页面数量正确。
2. 页面顺序正确。
3. 缺模板时跳过对应页面。
4. 每个页面携带正确上下文。

## 8. PaginationService

### 8.1 职责

计算实际页码和章节页码跨度。

### 8.2 输入

```ts
type PaginationInput = {
  pages: PagePlanItem[];
};
```

### 8.3 输出

```ts
type PaginationOutput = {
  pages: PagePlanItem[];
  chapterRanges: Record<number, string>;
};
```

### 8.4 规则

1. COVER 命名编号为 `00`。
2. TOC 实际页码为 `01`。
3. TOC 后页面依次递增。
4. `PAGE_PAGE_TEXT` 使用格式化实际页码，例如 `01`、`02`。
5. `TOC_PAGE_RANGE_TEXT` 使用章节跨度。
6. 章节跨度从 CHAPTER 页开始，到该章节最后页面结束。

### 8.5 测试点

1. 有 COVER + TOC。
2. 无 COVER 但有 TOC。
3. 单章节无 TITLE。
4. 章节含多个 TITLE 和 STEP。
5. 跨度格式为 `03-08`。

## 9. NamingService

### 9.1 职责

生成稳定、可读的 Frame 名称。

### 9.2 输入

```ts
type NamingInput = {
  page: PagePlanItem;
};
```

### 9.3 输出

```ts
type NamingOutput = {
  frameName: string;
};
```

### 9.4 规则

1. 名称包含实际页码或封面编号。
2. 名称包含页面类型。
3. 名称包含对应大纲文本。
4. 清理 Figma 节点名中的换行和多余空格。

### 9.5 测试点

1. COVER 名称。
2. TOC 名称。
3. CHAPTER 名称。
4. TITLE 名称。
5. STEP 名称。
6. 长文本截断策略是否稳定。

## 10. LayoutEngine

### 10.1 职责

根据页面计划和模板尺寸计算生成节点坐标。

### 10.2 输入

```ts
type LayoutInput = {
  pages: PagePlanItem[];
  templateSizes: Record<string, { width: number; height: number }>;
  options: {
    rowGap: number;
    stepGap: number;
    titleGroupGap: number;
  };
};
```

### 10.3 输出

```ts
type LayoutOutput = {
  placements: Record<string, { x: number; y: number }>;
};
```

### 10.4 规则

1. `rowGap = 100`。
2. `stepGap = 100`。
3. `titleGroupGap = 200`。
4. 每个章节一行。
5. STEP 跟随所属 TITLE。
6. COVER 和 TOC 放在最前面。

### 10.5 测试点

1. 单章节布局。
2. 多章节换行。
3. 同标题多个 STEP 间距 100。
4. 不同 TITLE 小队间距 200。
5. 不同尺寸模板不重叠。

## 11. GeneratedSectionManager

### 11.1 职责

删除旧生成 Section，创建新 Section，设置 pluginData，生成后选中新 Section。

### 11.2 输入

```ts
type SectionManagerInput = {
  currentPage: PageNode;
  sectionName: string;
};
```

### 11.3 输出

```ts
type SectionManagerOutput = {
  section: SectionNode;
  removedCount: number;
};
```

### 11.4 规则

1. 只删除 pluginData 标记为 `outline2page.generated=true` 的 Section。
2. 不删除模板。
3. 不删除普通用户节点。
4. 新 Section 写入 pluginData。
5. 生成完成后设置 `figma.currentPage.selection = [section]`。

### 11.5 测试点

1. 删除旧 Section。
2. 保留无标记节点。
3. 保留模板节点。
4. 新 Section 有 pluginData。
5. 生成后选中新 Section。

## 12. NodeFactory

### 12.1 职责

根据模板节点创建生成节点。

### 12.2 输入

```ts
type NodeFactoryInput = {
  page: PagePlanItem;
  template: TemplateInfo;
  section: SectionNode;
  placement: { x: number; y: number };
};
```

### 12.3 输出

```ts
type NodeFactoryOutput = {
  node: SceneNode | null;
  warnings: GenerationWarning[];
};
```

### 12.4 规则

1. Frame 使用 `clone()`。
2. Component 使用 `createInstance()`。
3. Instance 使用 `clone()`。
4. 其他节点尝试 `clone()`。
5. 设置节点名称为 PagePlanItem.frameName。
6. 设置节点坐标。
7. 节点追加到生成 Section。

### 12.5 测试点

1. Frame clone。
2. Component createInstance。
3. Instance clone。
4. 复制失败返回警告。
5. 名称和坐标正确。

## 13. PropertyInjector

### 13.1 职责

写入组件属性和普通 Text Layer。

### 13.2 输入

```ts
type PropertyInjectorInput = {
  node: SceneNode;
  page: PagePlanItem;
  chapterRanges: Record<number, string>;
};
```

### 13.3 输出

```ts
type PropertyInjectorOutput = {
  writtenCount: number;
  missingProperties: string[];
  warnings: InjectionWarning[];
};
```

### 13.4 规则

1. 递归扫描 node 内所有 InstanceNode。
2. 对存在的 componentProperties 完全匹配写入。
3. 递归扫描所有 Text Layer。
4. Text Layer 名称完全匹配时写入 characters。
5. 多个同名目标全部写入。
6. `TYPE` 不写。
7. 字体加载失败记录警告。

### 13.5 属性映射

| 目标 | 值 |
| --- | --- |
| `PAGE_TITLE_TEXT` | page.titleText |
| `PAGE_CHAPTER_TEXT` | page.chapterTitle |
| `PAGE_STEP_TEXT` | page.stepText |
| `PAGE_CHAPTER_TEXT (HUGE)` | page.chapterTitle |
| `PAGE_PAGE_TEXT` | page.pageNumberText |
| `PAGE_VERSION_TEXT` | page.vision |
| `TOC_NUM_TEXT` | chapterIndex |
| `TOC_PAGE_RANGE_TEXT` | chapterRanges[chapterIndex] |

### 13.6 测试点

1. 写入组件属性。
2. 写入 Text Layer。
3. 多个同名属性全部写入。
4. 缺属性继续。
5. TYPE 不被修改。

## 14. NavigationInjector

### 14.1 职责

写入导航栏项文案、显示状态和高亮状态。

### 14.2 输入

```ts
type NavigationInjectorInput = {
  node: SceneNode;
  page: PagePlanItem;
  document: OutlineDocument;
};
```

### 14.3 输出

```ts
type NavigationInjectorOutput = {
  showWrittenCount: number;
  highlightWrittenCount: number;
  warnings: NavigationWarning[];
};
```

### 14.4 导航节点识别

识别名称：

```text
NAV_CHAPTER
NAV_TITLE
NAV_STEP
```

显式序号：

```text
NAV_CHAPTER_01
NAV_TITLE_01
NAV_STEP_01
```

若模板没有后缀，插件根据大纲和视觉顺序赋予逻辑序号。

### 14.5 排序规则

1. 有 `_NN` 后缀时使用后缀。
2. 没有后缀时，同类节点按 `y` 从小到大、`x` 从小到大排序。
3. 排序结果第一个为 01。

### 14.6 写入规则

1. 对存在对应大纲项的导航节点写 `SHOW=true`。
2. 对超出大纲数量的导航节点写 `SHOW=false`。
3. 当前 CHAPTER / TITLE / STEP 写 `HIGHLIGHT=on`。
4. 非当前项写 `HIGHLIGHT=off`。
5. 无 STEP 页面不强行高亮 STEP。

### 14.7 测试点

1. 显式后缀排序。
2. 无后缀按视觉顺序排序。
3. 多余 STEP 隐藏。
4. 当前 STEP 高亮。
5. TITLE 页无 STEP 时只高亮 TITLE。

## 15. TocExpander

### 15.1 职责

扩展 TOC 页面中的 `NAV_group`，并注入目录数据。

### 15.2 输入

```ts
type TocExpanderInput = {
  tocNode: SceneNode;
  document: OutlineDocument;
  chapterRanges: Record<number, string>;
};
```

### 15.3 输出

```ts
type TocExpanderOutput = {
  expandedCount: number;
  writtenCount: number;
  warnings: TocWarning[];
};
```

### 15.4 规则

1. 查找名称为 `NAV_group` 的节点。
2. 按 TOC 所需行数判断是否复制。
3. 不够时复制最后一个 `NAV_group`。
4. 复制行按原间距排列。
5. 每行注入对应 CHAPTER / TITLE / NUM / RANGE。

### 15.5 测试点

1. NAV_group 足够。
2. NAV_group 不足时扩展。
3. 缺少 NAV_group 时警告。
4. 章节号写入正确。
5. 页码跨度写入正确。

## 16. ReportBuilder

### 16.1 职责

汇总所有模块输出，生成 UI 可读报告。

### 16.2 输入

```ts
type ReportBuilderInput = {
  createdCount: number;
  removedCount: number;
  skippedPages: SkippedPage[];
  warnings: AppWarning[];
  tocExpandedCount: number;
  selectedNodeIds: string[];
};
```

### 16.3 输出

```ts
type GenerationReport = {
  createdCount: number;
  replacedSectionCount: number;
  skippedCount: number;
  warningCount: number;
  warnings: AppWarning[];
  tocExpandedCount: number;
  selectedNodeIds: string[];
};
```

### 16.4 测试点

1. 报告计数正确。
2. 缺模板警告展示。
3. 缺属性警告展示。
4. TOC 扩展数量展示。

## 17. PluginUiBridge

### 17.1 职责

管理 UI 和插件主线程之间的消息。

### 17.2 消息

UI 发出：

```ts
SCAN_TEMPLATES
PARSE_OUTLINE
GENERATE
```

主线程返回：

```ts
TEMPLATES_SCANNED
OUTLINE_PARSED
GENERATION_DONE
ERROR
```

### 17.3 测试点

1. 打开插件自动扫描模板。
2. 粘贴 Markdown 后返回解析预览。
3. 点击生成后返回报告。
4. 主线程异常时 UI 展示错误。
