# Outline2Page 详细设计文档

## 1. 文档目的

本文档用于指导 Outline2Page Figma 插件的后续代码计划与模块拆分。本文不直接写实现代码，而是明确技术方案、数据结构、模块边界、节点处理策略、异常策略和可测试目标。

## 2. 已确认产品边界

1. 插件从零开始开发。
2. 技术栈使用 TypeScript + 原生 HTML/CSS/JS。
3. 用户在插件 UI 中直接粘贴 Markdown 大纲。
4. 插件只扫描当前 Figma Page。
5. 插件只在当前 Figma Page 生成结果。
6. 插件生成结果放入一个 Section 中。
7. 生成完成后选中新生成的 Section，方便用户手动移动调整。
8. 模板名称以 `PAGE_TEMP：` 开头。
9. 模板保留不动，不改名、不隐藏、不移动。
10. 模板可以是 Frame、Component、Instance 等可复制节点。
11. 模板复制策略为：Frame clone Frame；Component create Instance；Instance clone Instance。
12. 旧生成结果通过 Figma pluginData 内部标记识别。
13. 重新生成时删除旧的插件生成 Section，并创建新的 Section。
14. 用户额外加在生成 Section 外部的内容不受影响。
15. 用户手动改过插件生成 Section 内部内容，下次重新生成会被覆盖。
16. 插件同时写组件属性和普通 Text Layer。
17. 多个同名属性或文本层全部写入相同内容。
18. 导航项可通过命名 `NAV_CHAPTER`、`NAV_TITLE`、`NAV_STEP` 识别；插件可根据大纲顺序补充排序编号，例如 `NAV_STEP_01`。
19. `TYPE` 不由插件定义；用户通过模板手动定义好 TYPE 变体和暴露属性。
20. `PAGE_VERSION_TEXT` 注入 `《Vision》` 主题文本。
21. `PAGE_PAGE_TEXT` 注入实际页码。
22. `TOC_NUM_TEXT` 注入章节号。
23. `TOC_PAGE_RANGE_TEXT` 注入章节页码跨度。
24. `NAV_group` 是所有可复制导航行或目录行的容器命名。
25. 缺失模板时生成可生成部分，并报告缺失。
26. 缺失属性时继续生成，并报告警告。

## 3. 技术栈

### 3.1 插件主线程

使用 TypeScript 编写 Figma 插件主线程。

职责：

1. 扫描当前 Page 模板。
2. 读取和写入 Figma 节点。
3. 复制模板。
4. 创建 Section。
5. 写入组件属性和 Text Layer。
6. 设置 pluginData。
7. 返回生成报告。

### 3.2 插件 UI

使用原生 HTML/CSS/JS。

职责：

1. 粘贴 Markdown。
2. 展示解析预览。
3. 展示模板选择下拉栏。
4. 展示属性检测、警告、生成报告。
5. 与插件主线程通过 `postMessage` 通信。

### 3.3 构建方式

后续开发可使用轻量构建工具将 TypeScript 编译为 Figma 可运行的 `code.js`。UI 可先保持单文件 `ui.html`，降低开发复杂度。

本文档不指定具体打包工具，开发计划中仅要求满足：

1. TypeScript 可编译。
2. Figma manifest 可加载。
3. UI 与主线程可通信。
4. 单元测试可运行纯逻辑模块。

## 4. 核心流程

```text
打开插件
  ↓
扫描当前 Page 的 PAGE_TEMP：模板
  ↓
用户粘贴 Markdown
  ↓
UI 请求解析
  ↓
解析模块返回大纲树、统计、警告、所需页面类型
  ↓
UI 根据所需页面类型展示模板下拉栏
  ↓
用户选择模板
  ↓
点击生成
  ↓
页面计划模块生成 PagePlan
  ↓
覆盖模块删除旧生成 Section
  ↓
布局模块计算每个页面坐标
  ↓
节点生成模块复制模板并创建 Section
  ↓
属性注入模块写组件属性和 Text Layer
  ↓
导航模块写 SHOW / HIGHLIGHT / 文案
  ↓
TOC 模块扩展 NAV_group 并注入目录
  ↓
报告模块汇总结果
  ↓
选中新 Section
  ↓
UI 展示生成报告
```

## 6. Figma 节点策略

### 6.1 模板扫描

扫描范围为 `figma.currentPage`。

识别规则：

```text
node.name startsWith "PAGE_TEMP："
```

支持节点类型：

1. Frame
2. Component
3. Instance
4. Section 内部的上述节点
5. 其他可 clone 的 SceneNode

不支持复制的节点需要进入警告列表。

### 6.2 模板复制

复制规则：

| 模板类型 | 生成策略 |
| --- | --- |
| Frame | `clone()` |
| Component | `createInstance()` |
| Instance | `clone()` |
| 其他可复制节点 | `clone()`，失败则警告 |

生成节点需要追加到新创建的 Section 中。

### 6.3 Section 创建

每次生成创建一个新的 Section：

```text
Outline2Page_GENERATED
```

Section 使用 pluginData 标记：

```text
pluginData key: outline2page.generated
pluginData value: true
```

可以额外写入：

```text
outline2page.version
outline2page.generatedAt
outline2page.sourceHash
```

### 6.4 旧结果覆盖

重新生成前，在当前 Page 中查找 pluginData 标记为 `outline2page.generated=true` 的 Section。

只删除匹配 Section。

不删除：

1. `PAGE_TEMP：` 模板。
2. 没有 pluginData 标记的节点。
3. 用户放在生成 Section 外部的节点。

## 7. 页面计划规则

### 7.1 页码

封面页使用 `00` 命名编号，正文实际页码从 TOC 开始：

1. TOC 算第一页。
2. COVER 可不参与正文页码。
3. `PAGE_PAGE_TEXT` 注入格式化实际页码，例如 `01`、`02`。
4. TOC 页的实际页码为 `01`。

基础页码策略：

```text
COVER: 00
TOC: 01
第一个 CHAPTER: 02
后续页面依次递增
```

### 7.2 Frame 命名

```text
00_COVER_主题名
01_TOC
02_CHAPTER_章节名
03_TITLE_章节名_标题名
04_STEP_章节名_标题名_小节名
```

Frame 名称必须包含页面类型和对应大纲文本。

### 7.3 TOC 页码跨度

`TOC_PAGE_RANGE_TEXT` 表示章节对应页面跨度。

计算规则：

1. 章节页为该章节跨度起点。
2. 该章节最后一个 TITLE 或 STEP 页为跨度终点。
3. 如果章节只有 CHAPTER 页，则起点等于终点。
4. 格式为 `03-08`。

## 8. 布局规则

生成 Section 内部页面按章节分行。

间距：

1. 不同章节行垂直间距：`100px`。
2. 同一标题下 STEP 横向间距：`100px`。
3. 同一章节内不同 TITLE 小队横向间距：`200px`。

布局模块必须基于实际生成节点的 width / height 计算位置。

每个章节行的基本顺序：

```text
CHAPTER → TITLE → STEP → STEP → TITLE → STEP
```

COVER 和 TOC 位于最前方。它们可以作为第一行的前置页面，与后续章节行保持 `100px` 间距。

## 9. 属性注入规则

### 9.1 注入范围

对每个生成节点递归扫描：

1. InstanceNode 的 `componentProperties`。
2. 名称完全匹配属性名的 Text Layer。

### 9.2 组件属性写入

当 InstanceNode 包含目标属性时，调用属性写入能力设置值。

目标属性：

| 属性名 | 写入值 |
| --- | --- |
| `PAGE_TITLE_TEXT` | 当前 TITLE |
| `PAGE_CHAPTER_TEXT` | 当前 CHAPTER |
| `PAGE_STEP_TEXT` | 当前 STEP |
| `PAGE_CHAPTER_TEXT (HUGE)` | 当前 CHAPTER |
| `PAGE_PAGE_TEXT` | 格式化实际页码，`01`、`02` |
| `PAGE_VERSION_TEXT` | `《Vision》` |
| `TOC_TITLE_TEXT` | TOC 行标题 |
| `TOC_CHAPTER_TEXT` | TOC 行章节 |
| `TOC_NUM_TEXT` | 章节号，`01`、`02` |
| `TOC_PAGE_RANGE_TEXT` | 章节跨度，`03-08` |

### 9.3 Text Layer 写入

如果 Text Layer 名称完全等于目标属性名，同步写入相同文本。

写入 Text Layer 前必须确保字体可用。字体加载失败时，保留原文并记录警告。

### 9.4 TYPE

插件不主动定义或设置 `TYPE`。

原因：

1. 用户会人为定义 TYPE 值。
2. 不同 TYPE 决定模板暴露哪些属性。
3. 插件只负责匹配当前可见、已暴露的属性并写入。

## 10. 导航状态规则

### 10.1 导航项识别

导航项通过节点名称识别：

```text
NAV_CHAPTER
NAV_TITLE
NAV_STEP
```

如果模板没有 `_01` 后缀，插件根据节点在同级或同类集合中的顺序赋予逻辑序号。

排序规则：

1. 优先读取显式后缀：`NAV_STEP_01`、`NAV_STEP_02`。
2. 没有后缀时，按节点在画布上的视觉顺序排序。
3. 视觉顺序为：从上到下、从左到右。

### 10.2 SHOW

`SHOW` 写入布尔值：

```text
true / false
```

规则：

1. 大纲中存在对应序号导航项，写入 `true`。
2. 不存在对应序号导航项，写入 `false`。
3. 多余的导航项不删除，只隐藏。

### 10.3 HIGHLIGHT

`HIGHLIGHT` 写入字符串：

```text
on / off
```

规则：

1. 当前页面所属 CHAPTER 对应导航项为 `on`。
2. 当前页面所属 TITLE 对应导航项为 `on`。
3. 当前页面所属 STEP 对应导航项为 `on`。
4. 非当前项为 `off`。
5. 如果页面没有 STEP，则只对 CHAPTER 和 TITLE 做高亮判断。

## 11. TOC 扩展规则

TOC 中的可复制行或导航组命名为：

```text
NAV_group
```

规则：

1. 扫描 TOC 模板生成节点中的 `NAV_group`。
2. 根据 TOC 需要显示的章节与标题数量计算所需行数。
3. 如果现有 `NAV_group` 不够，复制最后一个 `NAV_group`。
4. 复制出来的行需要按顺序重新定位。
5. 自动扩展后在报告中写入扩展数量。

TOC 行数据：

1. Chapter 行写入 `TOC_CHAPTER_TEXT`。
2. Title 行写入 `TOC_TITLE_TEXT`。
3. `TOC_NUM_TEXT` 写入章节号。
4. `TOC_PAGE_RANGE_TEXT` 写入章节页面跨度。

## 12. UI 设计

UI 使用 Figma 原生风格。

核心区域：

1. 顶部标题：Outline2Page。
2. Markdown 输入区。
3. 解析按钮或自动解析。
4. 解析统计卡片。
5. 结构树预览。
6. 模板选择区。
7. 警告列表。
8. 生成按钮。
9. 生成报告。

UI 不负责直接操作 Figma 节点，所有节点操作通过主线程完成。

## 13. 消息通信

### 13.1 UI 到主线程

```ts
type UiToPluginMessage =
  | { type: "SCAN_TEMPLATES" }
  | { type: "PARSE_OUTLINE"; markdown: string }
  | { type: "GENERATE"; markdown: string; templateMapping: Record<string, string> };
```

### 13.2 主线程到 UI

```ts
type PluginToUiMessage =
  | { type: "TEMPLATES_SCANNED"; templates: TemplateInfo[]; warnings: string[] }
  | { type: "OUTLINE_PARSED"; document: OutlineDocument; summary: OutlineSummary }
  | { type: "GENERATION_DONE"; report: GenerationReport }
  | { type: "ERROR"; message: string; details?: string };
```

## 14. 错误策略

### 14.1 继续生成并警告

以下情况继续生成：

1. 缺少某些非当前页面所需属性。
2. Text Layer 字体加载失败。
3. Markdown 出现未定义层级。
4. 某些页面缺少模板。
5. TOC 行不足但可自动复制。

### 14.2 跳过部分生成

以下情况跳过对应页面：

1. 当前页面类型没有选择模板。
2. 选择的模板节点不存在。
3. 模板复制失败。

### 14.3 阻止生成

仅在没有任何可生成页面时阻止生成。

## 15. 测试原则

模块应尽量独立，优先把纯逻辑拆出 Figma API 依赖。

可独立测试模块：

1. Markdown 解析。
2. 页面计划。
3. 页码和章节跨度计算。
4. Frame 命名。
5. 布局坐标计算。
6. 导航序号排序。
7. 属性值映射。
8. 报告合并。

需要 Figma Mock 的模块：

1. 模板扫描。
2. 节点复制。
3. Section 创建。
4. componentProperties 写入。
5. Text Layer 写入。
