# Outline2Page 开发计划

## 1. 开发目标

本计划用于把 Outline2Page 从空项目推进到可在 Figma 中运行的插件原型，并为后续功能完善提供清晰顺序。

开发目标：

1. 从零搭建 TypeScript + 原生 HTML/CSS/JS Figma 插件工程。
2. 优先完成可单独测试的纯逻辑模块。
3. 再接入 Figma API 相关模块。
4. 最后完成 UI、集成、异常报告和验收测试。

## 2. 阶段划分

| 阶段 | 目标 | 主要产物 |
| --- | --- | --- |
| Phase 0 | 工程初始化 | manifest、package、tsconfig、基础构建 |
| Phase 1 | 纯逻辑核心 | Parser、Analyzer、Planner、Pagination、Layout |
| Phase 2 | Figma 节点适配 | 模板扫描、Section 管理、节点复制 |
| Phase 3 | 注入能力 | 属性注入、导航注入、TOC 扩展 |
| Phase 4 | UI 与通信 | 原生 UI、消息协议、报告展示 |
| Phase 5 | 集成验证 | Figma 手动验证、测试补齐、文档更新 |

## 3. Phase 0：工程初始化

### 3.1 任务

1. 创建 `package.json`。
2. 创建 `manifest.json`。
3. 创建 `tsconfig.json`。
4. 创建 `src/main.ts`。
5. 创建 `src/ui.html`。
6. 创建基础构建命令。
7. 创建测试框架。

### 3.2 建议依赖

基础开发依赖：

```text
typescript
@figma/plugin-typings
```

测试依赖可选择：

```text
vitest
```

是否引入测试框架在实现阶段确认，但模块设计应保持可测试。

### 3.3 验收标准

1. TypeScript 可以编译。
2. Figma 可以加载插件。
3. 打开插件能显示空白 UI 或基础 UI。
4. UI 能向主线程发送一条测试消息。
5. 主线程能返回一条测试消息。

## 4. Phase 1：纯逻辑核心

### 4.1 OutlineParser

任务：

1. 解析 Vision。
2. 解析 TOC。
3. 解析 CHAPTER。
4. 解析 TITLE。
5. 解析 STEP。
6. 记录层级异常警告。

验收：

1. 完整大纲可解析。
2. 缺少 Vision / TOC 有警告。
3. `####` 有警告但不中断。
4. TITLE / STEP 缺上级有警告。

### 4.2 OutlineAnalyzer

任务：

1. 统计章节数、标题数、小节数。
2. 计算预计总页数。
3. 推导 requiredPageKinds。

验收：

1. CHAPTER-only 大纲只要求 CHAPTER。
2. CHAPTER + TITLE 大纲要求 TITLE。
3. CHAPTER + TITLE + STEP 大纲要求 STEP。
4. 页数统计正确。

### 4.3 PagePlanner

任务：

1. 生成 COVER 页面计划。
2. 生成 TOC 页面计划。
3. 生成 CHAPTER / TITLE / STEP 页面计划。
4. 对缺模板页面生成 skippedPages。

验收：

1. 页面顺序符合大纲。
2. 缺模板不影响其他页面。
3. 每个 PagePlanItem 带完整上下文。

### 4.4 PaginationService

任务：

1. 设置 TOC 为实际页码 `01`。
2. 后续页面递增。
3. 生成章节页码跨度。
4. 为 `PAGE_PAGE_TEXT` 准备实际页码。

验收：

1. `PAGE_PAGE_TEXT` 可得到 `01`、`02`、`03`。
2. 章节跨度可得到 `03-08`。
3. 单页章节跨度可得到 `03-03`。

### 4.5 NamingService

任务：

1. 生成 COVER / TOC / CHAPTER / TITLE / STEP 名称。
2. 清理非法换行和多余空格。
3. 保持名称可读。

验收：

1. 名称包含类型。
2. 名称包含大纲文本。
3. 名称与页面顺序一致。

### 4.6 LayoutEngine

任务：

1. 根据模板尺寸计算坐标。
2. 按章节分行。
3. STEP 间距 100。
4. TITLE 小队间距 200。
5. 章节行间距 100。

验收：

1. 页面不重叠。
2. 多章节正确换行。
3. 多 STEP 跟随 TITLE。

## 5. Phase 2：Figma 节点适配

### 5.1 TemplateScanner

任务：

1. 扫描 `figma.currentPage`。
2. 找到 `PAGE_TEMP：` 模板。
3. 提取模板类型猜测。
4. 提取节点尺寸。
5. 提取组件属性名。
6. 提取 Text Layer 名称。

验收：

1. 只扫描当前 Page。
2. Frame / Component / Instance 都能识别。
3. 模板保留不动。
4. 不可复制节点有警告。

### 5.2 GeneratedSectionManager

任务：

1. 查找旧 Section。
2. 删除带 pluginData 标记的旧 Section。
3. 创建新 Section。
4. 设置 pluginData。
5. 生成完成后选中新 Section。

验收：

1. 只删除插件旧 Section。
2. 不删除模板。
3. 不删除用户外部内容。
4. 选中新 Section。

### 5.3 NodeFactory

任务：

1. Frame clone。
2. Component createInstance。
3. Instance clone。
4. 设置名称。
5. 设置坐标。
6. 放入 Section。

验收：

1. 不改变模板。
2. 生成节点类型符合策略。
3. 复制失败不中断全部流程。

## 6. Phase 3：注入能力

### 6.1 PropertyInjector

任务：

1. 写组件属性。
2. 写普通 Text Layer。
3. 多个同名目标全部写入。
4. 加载字体后写入文字。
5. 记录缺失属性警告。
6. 不修改 TYPE。

验收：

1. `PAGE_TITLE_TEXT` 正确。
2. `PAGE_CHAPTER_TEXT` 正确。
3. `PAGE_STEP_TEXT` 正确。
4. `PAGE_VERSION_TEXT` 写入 Vision。
5. `PAGE_PAGE_TEXT` 写入实际页码。
6. Text Layer 与组件属性同步。

### 6.2 NavigationInjector

任务：

1. 识别 `NAV_CHAPTER` / `NAV_TITLE` / `NAV_STEP`。
2. 识别显式后缀 `_01`。
3. 无后缀时按视觉顺序赋序号。
4. 写 `SHOW=true/false`。
5. 写 `HIGHLIGHT=on/off`。
6. 写导航文案。

验收：

1. 多余 STEP 隐藏。
2. 当前 STEP 高亮。
3. 当前 TITLE 高亮。
4. 无 STEP 页面不错误高亮 STEP。
5. 无后缀模板也能按顺序写入。

### 6.3 TocExpander

任务：

1. 查找 `NAV_group`。
2. 计算 TOC 所需行数。
3. 不足时复制最后一行。
4. 写入章节、标题、章节号、页码跨度。
5. 记录扩展数量。

验收：

1. TOC 展示 CHAPTER + TITLE。
2. `TOC_NUM_TEXT` 是章节号。
3. `TOC_PAGE_RANGE_TEXT` 是章节跨度。
4. 行数不足时自动扩展。
5. 缺少 `NAV_group` 有警告。

## 7. Phase 4：UI 与通信

### 7.1 UI 页面

任务：

1. 实现 Markdown 输入区。
2. 实现解析统计区。
3. 实现结构树预览。
4. 实现模板下拉选择区。
5. 实现警告列表。
6. 实现生成按钮。
7. 实现生成报告。

验收：

1. UI 风格接近 Figma 原生。
2. 用户能粘贴大纲。
3. 用户能看到所需模板。
4. 用户能选择模板。
5. 用户能看到生成报告。

### 7.2 消息通信

任务：

1. 打开插件自动发送 `SCAN_TEMPLATES`。
2. 粘贴或点击解析时发送 `PARSE_OUTLINE`。
3. 点击生成时发送 `GENERATE`。
4. 主线程返回扫描、解析、生成结果。
5. UI 展示错误。

验收：

1. 消息类型稳定。
2. 主线程异常不会让 UI 无响应。
3. 生成中按钮有 loading 状态。

## 8. Phase 5：集成验证

### 8.1 手动 Figma 验证

准备测试文件：

1. `PAGE_TEMP：COVER`
2. `PAGE_TEMP：TOC`
3. `PAGE_TEMP：CHAPTER`
4. `PAGE_TEMP：TITLE`
5. `PAGE_TEMP：STEP`

准备模板内容：

1. 至少一个组件属性。
2. 至少一个同名 Text Layer。
3. 至少一组 `NAV_group`。
4. 至少一组 `NAV_TITLE`。
5. 至少一组 `NAV_STEP`。

验收：

1. 能生成完整页面。
2. 能重新生成覆盖旧 Section。
3. 模板不被修改。
4. 选中新 Section。
5. 属性注入正确。
6. TOC 自动扩展正确。

### 8.2 文档更新

任务：

1. 根据实现修正设计文档。
2. 增加用户使用说明。
3. 增加模板命名规范说明。

## 9. 开发顺序建议

推荐顺序：

1. Phase 0 工程初始化。
2. OutlineParser。
3. OutlineAnalyzer。
4. PagePlanner。
5. PaginationService。
6. NamingService。
7. LayoutEngine。
8. TemplateScanner。
9. GeneratedSectionManager。
10. NodeFactory。
11. PropertyInjector。
12. NavigationInjector。
13. TocExpander。
14. UI。
15. 集成测试。

原因：

1. 先完成纯逻辑，反馈最快。
2. Figma API 模块依赖纯逻辑输出。
3. UI 最后接入，可以减少重复调整。

## 10. 风险点

### 10.1 组件属性写入限制

Figma 组件属性类型可能不同，写入值必须匹配属性类型。

处理：

1. 写入前检查属性类型。
2. 类型不匹配时记录警告。
3. 不阻止生成。

### 10.2 Text Layer 字体加载

修改 Text Layer 前需要加载字体。

处理：

1. 读取 Text Layer 当前字体。
2. 调用字体加载。
3. 加载失败记录警告。

### 10.3 导航项无显式后缀

模板没有 `_01` 后缀时，需要视觉排序。

处理：

1. 先按 y。
2. y 接近时按 x。
3. 生成报告中提示使用自动排序。

### 10.4 TOC 行复制布局

复制 `NAV_group` 后需要保持行间距。

处理：

1. 如果至少有两行，使用前两行的 y 差作为行距。
2. 如果只有一行，使用该行高度加默认间距。

## 11. 完成定义

插件开发完成需要满足：

1. 能在 Figma 中加载。
2. 能扫描当前 Page 模板。
3. 能解析 Markdown 并预览。
4. 能按模板生成 Section。
5. 能覆盖旧生成 Section。
6. 能注入组件属性和 Text Layer。
7. 能处理导航 SHOW / HIGHLIGHT。
8. 能扩展 TOC 的 `NAV_group`。
9. 能显示完整生成报告。
10. 纯逻辑模块有单元测试。
11. Figma API 模块有 Mock 测试或手动验证记录。
